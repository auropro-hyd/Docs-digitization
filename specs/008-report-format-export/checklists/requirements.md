# Requirements Checklist: Client-Aligned Compliance Report (Spec 008)

Used at PR-review time to confirm every spec'd requirement landed in code. Tick a box when implementation + test for that line are merged.

## Functional Requirements (from spec.md)

### Rule-centric table layout

- [ ] **FR-001** Five columns rendered in exact order: Question · Compliance · Evidence From Document · Detailed Evidence · Mitigation. No additional column visible.
- [ ] **FR-016** Question column shows the verbatim `AuditRule.text`; long text wraps with tooltip/expander.

### Three-state compliance taxonomy

- [ ] **FR-002a** `compliant` → "Compliant" (green check)
- [ ] **FR-002b** `non_compliant` → "Action Required" (orange warning)
- [ ] **FR-002c** `uncertain` OR `error` OR HITL `needs_review` → "Needs Attention" (amber attention)
- [ ] **FR-002d** `not_applicable` → row excluded
- [ ] **FR-014** Rules with `status="not_applicable"` excluded from the rendered output
- [ ] **FR-020** HITL `user_approved` flips a `non_compliant` rule's badge to "Compliant" in the report

### Compliant-row presentation (Akhilesh's deviation)

- [ ] **FR-003** Evidence From Document column is EMPTY for compliant rows
- [ ] **FR-004** Detailed Evidence column is a 2-3-sentence cross-page summary for compliant rows (not per-page list)
- [ ] **FR-006a** Mitigation column reads literal "Not Applicable" for compliant rows (matches reference verbatim)

### Non-compliant / Needs-attention row presentation

- [ ] **FR-005** Evidence From Document shows page references in reference syntax: `PAGE:N`, `PAGE:N, M, K`, `PAGE:N to M` (range when ≥3 contiguous)
- [ ] **FR-006b** Mitigation column carries 1-4-sentence actionable text (investigation / CAPA / training)
- [ ] **FR-015** Multiple findings on the same rule collapse to ONE row; page numbers are the union; mitigation is the strongest

### Score removal (export only)

- [ ] **FR-007** Zero rendered display of `overall_score`, `model_score`, `review_adjusted_score`, `score_decomposition`, `score_methodology`, per-agent score, category score in the **exported artifact** (PDF / HTML / Markdown). On-screen view keeps them (FR-011).
- [ ] Visual check: download the PDF, run `pdftotext`, grep for `score|Score|/100` → no matches.

### Header / footer (reference fidelity)

- [ ] **FR-008a** Header band: generic compliance-suite logo top-left
- [ ] **FR-008b** Product name centered in header band — default "BMR Compliance Intelligence Suite", configurable via `AT_COMPLIANCE__REPORT_PRODUCT_NAME`
- [ ] **FR-008c** "TITLE OF DOCUMENT" label top-left
- [ ] **FR-008d** "Document is Draft" top-right
- [ ] **FR-008e** Metadata table (Document / Product / Batch No / Date Of Validation)
- [ ] **FR-009** Footer on every page: "Disclaimer Note : This document is electronically generated by Pharmix AI Printed By: {operator} Printed On: {timestamp}"

### Export formats

- [ ] **FR-010** `format=pdf` is the default and primary deliverable; HTML and Markdown remain as fallback formats

### On-screen view (rule table + score top section)

- [ ] **FR-011a** `/compliance?doc=X` renders the new rule-centric `RuleTable` (matching FR-001…FR-006, FR-014…FR-017) in place of the previous flat `FindingsTable`
- [ ] **FR-011b** `AgentScorecard` panels STAY at the top of the on-screen view (score info preserved for internal HITL workflow)
- [ ] **FR-011c** `ExecutiveSummary` block STAYS at the top of the on-screen view
- [ ] **FR-011d** Downloading the PDF / HTML from the same screen produces an artifact with NO scores anywhere (FR-007). Two surfaces, two policies.
- [ ] **FR-012** Inline HITL controls (approve / reject / modify) per rule row; updates the badge in place

### Backward compatibility

- [ ] **FR-013** `compliance_result.json` shape unchanged; existing fields preserved; pre-feature files load without migration
- [ ] **FR-018** `ComplianceFinding.mitigation_text` is an additive field defaulting to `""`; LLM-synth result persists here

### Multi-agent

- [ ] **FR-017** Same `rule_id` evaluated by multiple agents (cross-doc reconciliation) produces separate rows, agent chip on each

### Telemetry

- [ ] **FR-019** `compliance.report_rendered` event per export carrying `format`, `row_count`, `compliant_count`, `action_required_count`, `needs_attention_count`, `render_duration_ms`, `agent_filter`

## Edge cases (from spec.md)

- [ ] Rule with zero findings AND status=`not_applicable` → row excluded
- [ ] Rule with mixed findings across pages, some compliant some uncertain → row's status is the worst; pages are the union
- [ ] Rule with `error` status → bucketed as `Needs Attention`; mitigation indicates "Re-run the evaluation"
- [ ] Same `rule_id` across multiple agents → separate rows with agent chip
- [ ] Very long rule text (300+ chars) → truncated 2-line + tooltip
- [ ] 30+ pages in findings → renders as `PAGE:N to M` ranges
- [ ] Mitigation missing AND can't be synthesised → fallback "Review and remediate. Initiate a CAPA if the gap persists."
- [ ] PDF render fails → HTML fallback with `X-Render-Fallback: html` header
- [ ] Doc with zero applicable rules → metadata header + single "No applicable rules evaluated" row

## Success criteria (from spec.md)

- [ ] **SC-001** New operator can produce client-shareable PDF in ≤ 60 seconds from "compliance run done" to "PDF downloaded"
- [ ] **SC-002** Visual diff against reference PDF passes 9-of-10 layout checks
- [ ] **SC-003** Round-trip equality: every non-applicable rule in the JSON appears in the PDF text
- [ ] **SC-004** Two consecutive exports produce byte-identical PDFs (modulo the timestamp). HITL overrides correctly reflected
- [ ] **SC-005** 100% of `Action Required` and `Needs Attention` rows have non-empty mitigation
- [ ] **SC-006** Per-export LLM spend ≤ $0.50; mitigation cache persists across exports
- [ ] **SC-007** Existing `tests/compliance/` suite passes unchanged

## Out-of-scope confirmations

- [ ] No evaluation-time changes (this is a rendering feature only)
- [ ] No DOCX / XLSX export
- [ ] No multi-language mitigation
- [ ] No bulk multi-doc export

- [ ] Principle II (Staged Pipeline) — post-pipeline; no impact on segmentation / OCR / evaluation
